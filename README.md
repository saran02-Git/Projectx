Brain Tasks App – CI/CD Deployment using AWS EKS

Project Overview:

This project demonstrates a complete CI/CD pipeline to build, containerize, and deploy a web application to a Kubernetes cluster using AWS services.

This project uses a pre-built dist application provided for DevOps practice. The focus of this repository is on CI/CD pipeline implementation, containerization, and Kubernetes deployment rather than application development.

Architecture:

GitHub → CodePipeline → CodeBuild → DockerHub → Amazon EKS → LoadBalancer

* Source Code          : GitHub
* CI/CD                : AWS CodePipeline
* Build & Docker       : AWS CodeBuild
* Container Registry   : DockerHub
* Orchestration        : Amazon EKS (Kubernetes)
* Deployment           : Kubernetes Deployment & Service (LoadBalancer)

Workflow:

1.Code is pushed to GitHub repository.
2.CodePipeline triggers automatically.
3.CodeBuild:
            * Builds Docker image
            * Tags and pushes image to DockerHub
4.Kubernetes deployment:
                       * Updates cluster using kubectl
                    * Applies deployment.yml and service.yaml
5.Application is exposed via AWS LoadBalancer.

Technologies Used:

* AWS CodePipeline
* AWS CodeBuild
* Amazon EKS
* Docker
* Kubernetes
* GitHub

Deployment Files:

* buildspec.yml – CI/CD build and deployment instructions
* deployment.yml – Kubernetes Deployment configuration
* service.yaml – Kubernetes Service (LoadBalancer)

Challenges Faced:
                  This project involved extensive debugging and troubleshooting across multiple layers of AWS and Kubernetes integration.

Key challenges included:

* IAM role authentication issues between CodeBuild and EKS
* Misconfigured aws-auth ConfigMap
* Kubernetes RBAC permission errors
* CodeBuild not properly assuming IAM roles
* kubectl authentication failures inside the build environment
* Understanding the difference between IAM role ARN and STS assumed-role identity
* Resolving “forbidden” errors for Kubernetes resources

During the process, the pipeline failed multiple times due to different issues at each stage. In total, the CodeBuild process was executed approximately 83 times while identifying and fixing these problems. Each failure helped isolate a new issue, eventually leading to a complete and working CI/CD pipeline with proper authentication, authorization, and deployment flow.

Outcome

* Fully functional CI/CD pipeline
* Successful Docker image build and push
* Kubernetes deployment automated through pipeline
* Application successfully running and accessible via LoadBalance

About the Application Files:

The dist/ folder contains production-ready static files including HTML, CSS, JavaScript, and assets.These files are pre-built and ready for deployment. No build tools or package dependencies are required, as the focus of this project is purely on deployment and DevOps practices.

Conclusion:
          This project reflects hands-on experience in building and debugging a real-world CI/CD pipeline using AWS and Kubernetes. It highlights practical problem-solving across IAM, authentication, and deployment layers, resulting in a stable and production-style deployment.
